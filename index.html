Agile Methodology — The Basics Foundation

What is Agile? Agile is a project management and software development approach that delivers work in small, frequent increments instead of one big release. It's not just a process — it's a mindset focused on flexibility, collaboration, and customer satisfaction. Used by Google, Facebook, Amazon.

❓ Why Agile? — The Core Problem

Imagine a Food Ordering App. You build it using Waterfall. Midway, the customer says: "Add live tracking, add wallet, add ratings." In Waterfall you'd have to redo everything = costly and slow. Agile solves this by welcoming change at any stage.

Requirements change frequently Market changes rapidly Late changes are costly in Waterfall

⚙️ 12 Agile Principles

1Customer Satisfaction — Highest priority: early & continuous delivery of valuable software.
2Welcome Change — Even late in development, changing requirements = competitive advantage.
3Frequent Delivery — Deliver working software every few weeks to months. Prefer shorter timescale.
4Daily Collaboration — Business people and developers work together daily.
5Motivated Individuals — Build around motivated people; give support and trust.
6Face-to-Face Communication — Most efficient method of conveying information.
7Working Software = Progress — Primary measure of progress is working software.
8Sustainable Pace — Sponsors, developers, users maintain constant pace indefinitely.
9Technical Excellence — Continuous attention to technical excellence enhances agility.
10Simplicity — Art of maximising work NOT done is essential.
11Self-Organising Teams — Best architectures emerge from self-organising teams.
12Regular Reflection — Team reflects regularly and adjusts behaviour accordingly.

Extreme Programming (XP) Framework

What is XP? Proposed by Kent Beck in 1999. One of the most important Agile frameworks. Uses OOP approach. The idea: "If a practice is good, do it all the time — take it to the EXTREME." Uses 4 activities: Planning → Design → Coding → Testing.

💎 XP's 5 Values

📢 CommunicationEnhance communication among team members and with customers
🎯 SimplicityDesign only for immediate needs, not future assumptions
💬 FeedbackFrom 3 sources: software itself, customer, team members
💪 CourageDiscipline to design for today; accept future may change drastically
🤝 RespectAmong team members, stakeholders, and for the software itself
Memory: CSFCR — Communication, Simplicity, Feedback, Courage, Respect

🔄 XP's 4 Framework Activities

1. Planning

Gather requirements, user stories

2. Design

KIS principle, CRC cards

3. Coding

Pair programming, TDD

4. Testing

Unit, acceptance, regression

📋 Phase 1: Planning (7 Steps)

S1Listening — Developers understand customer needs, identify major outputs & system functionality.
S2User Stories — Requirements written as user stories by customer. Each story = feature + functionality + expected output. Stored on index cards.
S3Story Value Assignment — Customer assigns business value/priority to each story.
S4Story Cost Estimation — Dev team estimates cost in development weeks. If a story > 3 weeks → divide it into smaller stories.
S5Release Planning — Customer and devs decide stories for next release, delivery date, scope. Ordering strategies: all immediately / highest-value first / highest-risk first.
S6Project Velocity = Number of customer stories completed in a release. Used to estimate future schedules & detect over-commitment.
S7Continuous Planning — During development, customer may add/modify/split/remove stories. XP team revises release plans.

🎨 Phase 2: Design

XP follows the KIS Principle — Keep It Simple! Design only what is required NOW. No unnecessary features. Minimal documentation. Design evolves.
CRC Cards Primary XP design artifact.
C = Classes
R = Responsibilities
C = Collaborators
Spike Solution When design is difficult:
Quick prototype created → evaluated for feasibility → reduces risk
Refactoring Improve code structure without changing behaviour. Improves design quality, simplifies structure, reduces defects.

💻 Phase 3: Coding

Test-First Development (TDD) Write unit tests BEFORE coding. Write only code needed to pass tests.
Pair Programming Driver: writes code
Navigator: reviews & guides
→ Real-time review, knowledge sharing
Coding Standards Common conventions, collective code ownership, improved readability.
Continuous Integration Code integrated daily. Automated builds & tests. Early error detection.

🧪 Phase 4: Testing

Test TypeWhenPurpose
Unit TestingCreated BEFORE codingImmediate feedback on individual units
Regression TestingAfter every refactorEnsure changes don't break existing code
Integration TestingDailyCombine unit tests, test full system
Acceptance TestingAfter sprintDefined by customer, validates user stories

👍 Advantages of XP

  • High software quality
  • Early defect detection
  • Handles changing requirements
  • Continuous customer involvement

👎 Limitations of XP

  • Requires experienced developers
  • Requires customer availability
  • Pair programming culture may be difficult
  • Less suitable for large distributed teams

Scrum Most Popular Framework

What is Scrum? An Agile framework for developing complex products in uncertain, changing environments. Provides transparency (to team, customer, management). Used by Google, Microsoft, IBM, Yahoo!, TCS, Wipro, Infosys, Nokia, etc.

🏗️ Scrum Framework — 3 Pillars

👥 ROLES (3) • Product Owner
• Scrum Master
• Development Team
📅 EVENTS (4) • Sprint Planning Meeting
• Daily Scrum
• Sprint Review
• Sprint Retrospective
📦 ARTIFACTS (3) • Product Backlog
• Sprint Backlog
• Burndown Chart

👥 Scrum Roles in Detail

🎯
Product Owner
Owns the product vision. Gets input from all stakeholders. Creates and maintains the Product Backlog — single prioritised list of features, based on business value and risk. Can add, remove, reorder items.
🛡️
Scrum Master
Serves the team (removes impediments), Protects the team (from outside disruptions), Teaches Scrum. Without ScrumMaster → high risk of failure.
👨‍💻
Development Team
Ideal size: 7 ± 2 people. Cross-functional (designers + coders + testers). Self-organizing and self-managing. Everyone contributes based on competency.

📦 Scrum Artifacts in Detail

Product Backlog — Single master list of ALL features required. Prioritised by Product Owner based on business value + risk. Constantly revised. Items at top = done soonest.
Sprint Backlog — Tasks selected for THIS sprint + hours remaining for each task. Updated daily. Tracks progress during the sprint.
Burndown Chart — Graph showing total remaining work (hours) each day. Goes DOWN as work is completed. Helps team monitor sprint progress. If line is above ideal → team is behind.

📊 Example Sprint Backlog:

TaskInitialMonTueWedThuFri
Code UI16h1612840
Code middle tier8h86420
Test middle tier12h1212840
Write online help4h44200

⚡ The Sprint — Heart of Scrum

  • Fixed time period: typically 1–4 weeks (recommend starting with 2 weeks)
  • Sprints occur one after another — no down-time between sprints
  • During the sprint: NO CHANGES to what was committed or end-date
  • If something major comes up, Product Owner can terminate the sprint and start a new one
  • Goal: produce a Potentially Shippable Product — fully designed, implemented, and tested
Key rule: Sprint = Fixed commitment + Fixed time + No change in scope. This gives the team FOCUS and STABILITY.

📅 Scrum Events in Detail

1. Sprint Planning Meeting
Before each sprint. Team selects items from top of Product Backlog. Creates task-level plan. Estimates total hours vs available hours. Everyone participates regardless of experience. Important: Product Owner must NOT pressure team to over-commit.
2. Daily Scrum (Stand-up)
Every day, <15 minutes. Standing up (to keep it fast). Each person answers 3 things only:
① What did I do yesterday?  ② What will I do today?  ③ Any blockers?
ScrumMaster notes blocks and resolves them afterwards. Others can attend but cannot speak.
3. Sprint Review
End of sprint. Product Owner + Team + ScrumMaster + Stakeholders. Demo of what was built. Product Owner gathers feedback → incorporated back into Product Backlog.
4. Sprint Retrospective
After Sprint Review. Team + PO + ScrumMaster. Review HOW they worked (process). Identify improvements. Mechanism for continuous improvement. Surface critical problems to management.

Adaptive Software Development (ASD) Framework

What is ASD? Proposed by Jim Highsmith. Agile methodology for complex and uncertain software systems. Focuses on adaptation, collaboration, and learning. Software is developed in repeated cycles (increments). ASD focuses on adapting to change instead of following rigid plans.

🔑 Key Ideas of ASD

  • Emphasises human collaboration
  • Encourages self-organising teams
  • Supports continuous learning
  • Accepts changing requirements (like RAD)
ASD focuses on adapting to change instead of following rigid plans. The team learns from every cycle.

🔁 ASD Life Cycle — 3 Phases (REPEATING CYCLE)

Speculate

Replaces traditional planning. High-level adaptive cycle planning.

Collaborate

Team works together to build software.

Learn

Evaluate results, gather feedback, improve.

Memory: S-C-L → Speculate, Collaborate, Learn. Then repeat!
Phase 1: Speculation
Replaces traditional detailed planning. Uses: customer's mission statement + project constraints + basic requirements → defines set of release cycles (time-boxed plan). Purpose: plan at high level, allow adjustments as project progresses.
Phase 2: Collaboration
Team develops and tests software components. Key aspects: strong communication, teamwork & trust, customer interaction, shared problem solving.
Phase 3: Learning
Activities: Focus groups (customer feedback on increment), Technical reviews (design/implementation), Project postmortems (what worked, what didn't). Purpose: improve understanding + adjust plans for next cycle.

👍 Advantages of ASD

  • Handles changing requirements effectively
  • Encourages team collaboration
  • Supports continuous improvement
  • Reduces project risks (incremental)
  • Suitable for complex projects

📌 When to Use ASD

  • Requirements are uncertain or evolving
  • Projects are complex (research-based)
  • Continuous learning and adaptation required

Dynamic Systems Development Method (DSDM) Framework

What is DSDM? Agile approach that: provides a framework for building & maintaining systems, meets tight time constraints, uses incremental prototyping, operates in controlled project environment. Maintained by the DSDM Consortium (www.dsdm.org).

📐 DSDM Philosophy — The Modified Pareto Principle (80/20 Rule)

Core Idea: 80% of application functionality can be delivered in 20% of total development time. Focus on most important features first. Remaining features added in later iterations.

📦 E-commerce Example:

First 20% time delivers (80% value): User login, Product catalog, Add to cart, Checkout
Later iterations add: Product recommendations, Advanced search, Analytics dashboards
Key insight: In each iteration, only enough work is done to move to next increment. Remaining details completed later when more requirements are known.

🔄 DSDM Life Cycle — Phases

PhaseDescription
Pre-ProjectEnsures only right projects are started. Evaluates suitability. Prepares environment.
Feasibility StudyEstablishes basic business requirements. Identifies constraints. Determines if project is viable for DSDM.
Business StudyIdentifies functional + information requirements. Defines application architecture. Identifies maintainability requirements. Goal: ensure business value.
Functional Model IterationProduces incremental prototypes. Demonstrates system to customers. Gathers additional requirements. Prototypes → requirement validation.
Design and Build IterationRevisits prototypes. Ensures they're properly engineered. May run concurrently with Functional Model Iteration.
ImplementationPlaces latest software increment into operational environment. May not be 100% complete. Users may request changes → returns to Functional Model Iteration.
Post-ProjectSystem maintenance. Supporting operational environment. Future changes/improvements.
Memory: Pre–FS–BS–FMI–DBI–Impl–Post

🔗 Integration with Other Agile Methods

DSDM can combine with: XP (for technical practices) and ASD (for collaboration and self-organising team concepts).

Feature Driven Development (FDD) Framework

What is FDD? Proposed by Peter Coad, extended by Stephen Palmer and John Felsing. Practical process model for OOP software engineering. Adaptive agile process. Suitable for moderately large and complex software projects.

💡 Philosophy of FDD — 3 Key Ideas

1Collaboration — Team members collaborate closely.
2Feature-Based Decomposition — Manage complexity using feature decomposition, then integrate software increments.
3Multiple Communication Methods — Verbal, graphical, and text-based methods.

🌟 What is a "Feature" in FDD?

Feature = A client-valued function that can be implemented in two weeks or less.
Template format: <action> the <result> <by|for|of|to> a(n) <object>
Example Feature"Add the product to shopping cart"
Example Feature"Display the technical specifications of the product"
Example Feature"Store the shipping information for the customer"

Feature Set = Group of related features. Template: "<action>-ing a(n) <object>". Example: "Making a product sale" → includes add-to-cart, store shipping info, display product details.

👥 FDD Roles

📊
Project Manager
Overall project planning and progress tracking. Ensures features completed on schedule.
🏛️
Chief Architect
Designs overall system architecture. Guides team during domain modeling.
🗂️
Development Manager
Manages development team. Ensures coding standards and processes followed.
👨‍💻
Chief Programmer
Leads the feature team for a specific feature. Coordinates design and development work.
👥
Feature Team
Temporary team formed to develop a feature. Includes class owners of relevant classes.
🔤
Class Owner
Each class has one responsible developer. That dev maintains and updates that class.
🏢
Domain Expert
Provides business knowledge. Helps developers understand problem domain.

🔄 FDD's 5 Development Processes

P1Develop an Overall Model — Understand project domain. Small groups create domain models → merged into one overall system model. (Object model showing Customer, Product, Order, Payment)
P2Build a Feature List — Create list of features from domain model. Each feature ≤ 2 weeks. Large features divided into smaller ones.
P3Plan by Feature — Prioritise features. Assign work to developers. Distribute feature sets. Plan based on complexity and dependencies.
P4Design by Feature — Create detailed design (sequence diagrams). Refine system model. Design inspection with team. Each iteration typically = 2 weeks.
P5Build by Feature — Implement the feature. Code inspection. Feature tested. Integrated into main build. Features > 2 weeks → break into smaller features.
Memory: D-B-P-D-B → Develop model, Build feature list, Plan, Design, Build. (The last 2 repeat per feature.)

📊 FDD Project Management — 6 Milestones

① Design Walkthrough ② Design ③ Design Inspection ④ Code ⑤ Code Inspection ⑥ Promote to Build

🏆 Quality Assurance in FDD

  • Incremental development strategy
  • Design inspections
  • Code inspections
  • Software quality assurance audits
  • Metrics collection
  • Use of patterns for analysis, design, and construction

✅ FDD Best Practices

Domain object modelingDeveloping by feature Individual class ownershipFeature teams InspectionsConfiguration management Regular buildsProgress visibility

🛒 Example: Online Shopping System in FDD

ProcessWhat Happens
1. Overall ModelIdentify objects: Customer, Product, Order, Payment. Create domain class diagram. Customer places Order → Order contains Product → Order uses Payment.
2. Feature ListRegister customer, Login customer, Search product, Display product detail, Add product to cart, Place order, Make payment
3. Plan by FeatureRegister customer → Dev A; Search product → Dev B; Add to cart → Dev C; Payment → Dev D
4. Design by Feature"Add to cart" → Identify classes (Cart, Product, Customer). Customer selects product → System adds to cart → Cart updates total items. Create sequence diagram.
5. Build by FeatureDeveloper writes code → Code inspection → Feature tested → Added to main build.

Lean Software Development (LSD) Framework

What is LSD? Agile approach derived from lean manufacturing concepts (originated from Toyota Production System). Focuses on improving efficiency, reducing waste, and delivering high-value software quickly. Uses Minimum Viable Product (MVP) approach. Goal: produce MORE value with FEWER resources.

⚙️ 7 Key Principles of LSD

1Eliminate Waste — Remove anything that doesn't add value to the customer or product.
2Build Quality In — Quality built during development, not tested in afterwards. Use testing, code reviews, continuous integration.
3Amplify Learning — Software development is a learning process. Code reviews, meetings, pair programming increase knowledge.
4Defer Decisions (Decide as Late as Possible) — Decisions based on facts, not assumptions. Keep options open. Adapt to change.
5Deliver Fast — Release MVP quickly. Frequent releases. Speed reduces waste, increases customer satisfaction.
6Empower the Team (Respect) — Developers empowered, not strictly controlled. Collaboration, communication. Team participates in decisions.
7Optimise the Whole — Focus on improving entire system, not just parts. Avoid optimising one component at cost of overall performance.
Memory: EW-BQ-AL-DD-DF-ET-OW — Eliminate Waste, Build Quality, Amplify Learning, Defer Decisions, Deliver Fast, Empower Team, Optimize Whole

🗑️ 7 Types of Waste (to ELIMINATE)

1
Partially Done WorkWork started but not completed. E.g., Code waiting for testing.
2
Extra ProcessesWork that doesn't add value. E.g., Unused documentation, unnecessary approvals.
3
Extra FeaturesFeatures not required or "nice-to-have" only. Also called gold plating.
4
Task SwitchingMultitasking between projects = context-switching penalty. E.g., working on multiple projects.
5
WaitingDelays from waiting for reviews, approvals. E.g., waiting for prototype review.
6
MotionEffort to communicate between teams. E.g., distributed teams, handoffs.
7
DefectsErrors requiring correction. E.g., requirement defects, software bugs.
Memory: PP-EP-EF-TS-W-M-D → Partially done, Extra process, Extra feature, Task switching, Waiting, Motion, Defects

🍕 Example: Food Delivery App — LSD Applied

PrincipleHow Applied
1. Eliminate WasteSkip restaurant chat, loyalty games, advanced analytics at start. Focus only on login, menu browsing, order, payment.
2. Build Quality InTest if orders placed correctly and payments processed safely BEFORE release.
3. Amplify LearningUsers say live tracking is important → add it in next version.
4. Defer DecisionsUse simple payment gateway first; decide on multiple options later.
5. Deliver FastRelease MVP: order food from nearby restaurants, even without advanced features.
6. Respect the TeamDeveloper suggests improving checkout → team implements it.
7. Optimise the WholeImprove not just the app, but restaurant response time and delivery coordination too.
